# 智能姿态识别手套迁移方案（STM32F407ZGT6 → Renesas RA6M5）

## 1. 项目目标

本项目拟将原本运行在 **STM32F407ZGT6** 平台上的智能姿态识别手套功能，按“**前端采集 + 后端识别**”的思路迁移到 **Renesas RA6M5** 平台上。

考虑到比赛指定使用的 RA6M5 开发板体积较大，不适合直接安装在手套本体上，因此采用如下分工方案：

- **STM32F407ZGT6**：保留在手套端，负责传感器数据采集、姿态解算、特征整理与串口发送。
- **RA6M5**：作为主处理器，负责串口接收、特征解析、SVM 支持向量机分类、识别结果输出与后续控制扩展。

这种方案的优势是：

1. 不需要大改手套现有硬件结构；
2. 可最大程度复用 STM32 端已经验证过的采集链路；
3. 能突出 RA6M5 在算法处理和系统控制中的核心作用；
4. 调试边界清晰，便于比赛前快速稳定落地。

---

## 2. 系统总体架构

### 2.1 原系统架构

原有系统主要流程如下：

1. 读取手套各个手指和手背上的姿态传感器数据；
2. 进行角度解算和滤波；
3. 提取姿态特征向量；
4. 输入 SVM 分类器进行姿态识别；
5. 输出识别结果或控制命令。

### 2.2 迁移后系统架构

迁移后建议采用如下结构：

```text
MPU6050 / JY901S 等传感器
          ↓
      STM32F407ZGT6
（采集 / 滤波 / 特征整理 / 串口发送）
          ↓ UART
        RA6M5
（串口接收 / 特征解析 / SVM分类 / 结果输出）
          ↓
 上位机 / 屏幕 / 机械臂 / 小车 / 其他执行机构
```

## 3. 迁移设计思想

本次迁移不建议一开始就把全部底层驱动（如 MPU6050、I2C 多路复用、JY901S 解析等）全部移植到 RA6M5。

更稳妥的策略是：

### 第一阶段：先迁移“识别功能”

由 STM32 保留以下功能：

- 传感器采集；
- 姿态角解算；
- 滤波；
- 特征向量生成；
- 通过串口向 RA6M5 发送特征数据。

由 RA6M5 实现以下功能：

- 串口接收数据帧；
- 解析特征；
- 运行 SVM 模型；
- 输出识别结果。

### 第二阶段：后续增强

当第一阶段功能稳定后，再考虑继续把以下内容迁到 RA6M5：

- 原始加速度/角速度数据处理；
- 滤波与角度解算；
- 更复杂的多传感器融合逻辑。

---

## 4. 迁移任务总路线

推荐按以下顺序推进整个项目：

### Step 1：冻结 STM32 端采集逻辑

当前 STM32 上已经跑通的内容先不要改动，包括：

- MPU6050 数据采集；
- 多路传感器轮询；
- JY901S 数据读取；
- 角度滤波；
- 姿态特征提取。

只新增一个功能：

> 将原本送入 STM32 本地 SVM 的特征数据，改为通过串口发送到 RA6M5。

这样 STM32 就从“完整识别主控”转变为“手套前端数据采集板”。

---

### Step 2：确定 STM32 → RA6M5 串口通信协议

为了后续调试稳定，建议优先采用 **固定长度二进制帧协议**，而不是字符串协议。

#### 2.1 建议的数据内容

优先发送已经整理好的 **10 维特征向量**：

- finger1_x
- finger1_y
- finger2_x
- finger2_y
- finger3_x
- finger3_y
- finger4_x
- finger4_y
- roll
- pitch

#### 2.2 推荐帧格式

```c
typedef struct
{
    uint8_t  sof1;        // 0xAA
    uint8_t  sof2;        // 0x55
    uint8_t  ver;         // 协议版本
    uint8_t  seq;         // 帧序号
    uint32_t tick_ms;     // 时间戳
    int16_t  feature[10]; // 10维特征，建议放大100或1000倍
    uint8_t  status;      // 状态字
    uint16_t crc16;       // CRC校验
} glove_frame_t;
```

#### 2.3 协议建议

- 帧头固定：`0xAA 0x55`
- 特征值采用 `int16_t`
- 浮点角度转定点数发送，例如 `12.34° -> 1234`
- 初始波特率建议：`115200`
- 后续如果吞吐不足，可提升到 `460800`

#### 2.4 为什么先发 10 维特征而不是原始传感器数据

原因如下：

1. 数据量更小；
2. RA6M5 端调试更简单；
3. 分类结果更容易与原系统进行对比；
4. 出现问题时能快速判断是“通信问题”还是“算法问题”。

---

### Step 3：在 RA6M5 上先完成串口收包功能

这一阶段 **先不要急着移植 SVM**，而是先让 RA6M5 做最小功能验证：

1. 初始化 UART；
2. 接收 STM32 发来的固定长度数据帧；
3. 校验帧头和 CRC；
4. 正确解析出 10 维特征；
5. 通过串口打印接收到的特征值。

#### 3.1 这一阶段的验收标准

满足以下条件即可认为通信链路打通：

- 帧序号连续递增；
- 10 维数据解析正确；
- 手指和腕部姿态变化能实时反映到接收数据中；
- 长时间运行不丢帧、不乱帧。

---

### Step 4：将 SVM 分类模块从 STM32 中抽离

这一步是核心任务。

目标是把原 STM32 工程中的 SVM 相关代码整理成 **纯 C 算法模块**，使其不再依赖 STM32 HAL 库或芯片专用接口。

#### 4.1 需要去除的依赖

应尽量删除或隔离以下依赖：

- `HAL_UART_xxx`
- `HAL_Delay()`
- `HAL_GetTick()`
- STM32 专用头文件
- 中断回调里直接耦合的算法调用逻辑

#### 4.2 建议保留的算法内容

- 特征归一化；
- SVM 推理函数；
- 标签映射；
- 稳定判决与抗抖逻辑；
- 原有推理限频机制。

#### 4.3 推荐目录结构

```text
algo/
├── svm_model.c
├── svm_model.h
├── gesture_classify.c
└── gesture_classify.h

app/
├── gesture_app.c
└── gesture_app.h
```

推荐统一封装为：

```c
int Gesture_Classify(const float feature[10]);
```

---

### Step 5：在 RA6M5 上恢复原识别流程

当 RA6M5 已经能正确接收 10 维特征后，就可以把移植后的 SVM 模块接入到主循环中。

建议继续保留原项目中“**非每帧都推理，而是按一定节奏推理**”的思路，例如每 5 帧推理一次，以降低系统开销并增强结果稳定性。

#### 5.1 推荐伪代码

```c
while (1)
{
    if (frame_ready)
    {
        frame_ready = 0;
        parse_feature_to_float();

        gr_tick++;
        if (gr_tick >= 5)
        {
            gr_tick = 0;
            label = Gesture_Classify(feature10);
            cmd = label_to_cmd(label);
            uart_send_cmd(cmd);
        }
    }
}
```

#### 5.2 阶段验收标准

- RA6M5 的分类结果与 STM32 原始分类结果基本一致；
- 手势切换时结果更新及时；
- 识别稳定，不出现明显抖动或频繁误判。

---

### Step 6：完成识别结果输出与整机联调

当 RA6M5 上分类已经稳定后，再继续完善结果输出功能。

#### 6.1 最简单版本

先通过串口打印：

```c
printf("label=%d cmd=0x%02X\r\n", label, cmd);
```

#### 6.2 展示/比赛版本

后续可将识别结果输出到：

- 上位机；
- 显示屏；
- 机械臂；
- 小车；
- 其他被控设备。

如果需要进一步优化串口发送效率，可在功能稳定后再引入 DMA 发送机制。

---

## 5. 推荐开发阶段划分

为了降低调试难度，建议将整个项目拆成 3 个阶段推进。

### 阶段 A：通信链路打通

目标：

- STM32 将 10 维特征通过 UART 发出；
- RA6M5 稳定接收并打印。

输出成果：

- 串口协议定型；
- 数据通信稳定。

### 阶段 B：算法迁移到 RA6M5

目标：

- 将 STM32 中的 SVM 模块迁移到 RA6M5；
- RA6M5 基于接收到的特征进行姿态识别。

输出成果：

- RA6M5 成为识别核心；
- 与原 STM32 分类结果进行一致性验证。

### 阶段 C：整机展示与功能增强

目标：

- RA6M5 输出识别命令；
- 联动执行机构或显示界面；
- 增加抗抖、稳定判决与状态切换逻辑。

输出成果：

- 可用于比赛展示的完整演示系统。

---

## 6. 当前方案的主要优势

采用“STM32 采集 + RA6M5 分类”的双板架构，有以下明显优势：

1. **硬件改动小**：手套现有传感器采集板无需重做；
2. **功能迁移清晰**：RA6M5 专注承担识别与控制核心任务；
3. **调试效率高**：采集问题与分类问题可以分开定位；
4. **落地风险低**：比赛前更容易做出稳定可展示的系统；
5. **后续扩展性强**：后续可以逐步把更多算法前处理迁移到 RA6M5。

---

## 7. 当前最小可落地实施方案

建议当前优先实现如下最小版本：

### STM32 端

- 保留传感器驱动与滤波；
- 提取 10 维特征；
- 通过 UART 定时发送特征帧。

### RA6M5 端

- 接收并解析串口数据；
- 还原 10 维特征；
- 执行 SVM 分类；
- 输出识别结果。

这样即可快速形成一版完整、稳定、可演示的迁移成果。

---

## 8. 下一步实际开发建议

接下来建议按以下顺序开展具体工作：

1. 整理 STM32 端当前 10 维特征定义；
2. 整理 STM32 当前 SVM 输入数组；
3. 编写 STM32 → RA6M5 串口发送帧；
4. 在 RA6M5 上完成串口接收与解析；
5. 提取并移植 SVM 相关源码；
6. 对比 RA6M5 与 STM32 的分类结果；
7. 完成输出控制与整机联调。

---

## 9. 结论

本项目不宜简单理解为“把 STM32 程序原封不动搬到 RA6M5 上”，而更适合按照系统重构的方式进行迁移：

- **STM32 负责手套端实时采集与预处理**；
- **RA6M5 负责姿态识别与主控输出**。

这是当前最符合比赛要求、工程风险最低、实现效率最高的迁移路径。待该版本稳定后，再逐步考虑更深层次的驱动和算法前处理迁移。
