## 3. 从现有工程得到的关键结论

### 3.1 手套端当前稳定语义

现有工程和文档已经明确：

- `0x05 -> rock`
- `0x06 -> left`
- `0x07 -> right`
- `0x08 -> up`
- `0x0A -> down`
- `0x69 -> one`
- `0x19 -> three`
- `0x02 / 0x00` 当前不应参与机械臂联动

### 3.2 现有固件输出链路

从 `hal_entry.c`、`gesture_service.c` 和根目录文档可以确认：

- RA 当前链路最终对外输出的是**原始命令字节**
- `uart7` 只发命令字节，不发文本调试信息
- 现有固件内部已经做了分类节拍和基础过滤

所以 PC 侧第一版最合适的方案不是去重新接 31 字节特征帧，也不是重新做 SVM，而是直接接“已经稳定下发的原始命令字节”。

## 6. 当前版本的关键实现选择

### 6.1 为什么默认用 MOVJXYZ

V1 默认使用 `MOVJXYZ`，理由：

- 对示例点位和保守步进更稳
- 对初次联调更友好
- 比 `MOVLXYZ` 更不容易因为直线轨迹约束导致不可达

如果后续确认现场需要严格直线轨迹，可以在 `config.yaml` 中改成 `MOVLXYZ`。

### 6.2 为什么 0x19 默认映射成安全位

用户要求中允许：

- `0x19`：返回安全位，或预留特殊功能

V1 为了更利于闭环验收，优先把它做成“返回安全位”。

### 6.3 为什么 0x05 默认只做日志占位

用户要求中明确允许：

- 如果当前没有真实夹爪控制能力，`0x05` 可以先实现成“日志占位 + 预留接口”

当前未掌握现场末端套件安装状态，因此默认值是 `placeholder`。如果现场确认安装了夹爪，可在 `config.yaml` 改成 `gripper`。

## 7. 本次采用的关键假设

### 7.1 已采用的合理假设

1. PC 侧串口收到的是**单字节原始命令流**，而不是 31 字节特征帧。
2. 第一版现场主要目标是稳定单步运动，因此默认速度比例取了偏保守值。
3. `safe_pose` 和 `soft_limits` 可以先给出保守默认值，但首次上机前必须现场复核。
4. `0x19` 当前优先用于返回安全位。
5. `0x05` 当前默认做占位动作。

### 7.2 这些假设带来的影响

- 程序已经能形成最小闭环
- 但正式上机前仍需你按现场安装姿态校正 `config.yaml`

## 8. 当前版本支持什么

- 官方 Python DLL API 方式控制 Dobot Magician
- 单条命令触发单步离散动作
- 启动初始化与回安全位
- Pose 日志
- 手套串口原始字节日志
- 基础去抖、节流和软限位
- 单命令测试模式
- dry-run 自检模式

## 9. 当前版本还不支持什么

- 连续实时拟合跟随
- 多线程命令队列调度
- 完整报警码解析
- 更强健的掉线恢复
- GUI
- 对 `0x05` 的真实末端工装自适配识别
- `X` 负方向动作

## 11. 下一步最值得做什么

建议优先顺序如下：

1. 先在现场把 `safe_pose`、`soft_limits`、`serial.port`、`dobot.port` 调准。
2. 用 `--single-command` 单独验证每个动作方向。
3. 现场确认 `0x05` 是否需要切到真实夹爪模式。
4. 如果单步闭环稳定，再考虑给 PC 侧补一个小型状态面板或串口监视界面。
5. 再下一步才值得考虑 RA8P1 / 视觉 / 力觉 / 更复杂轨迹控制。
