# 工程大纲

## 1. 工程目标

当前工程的目标已经从“单纯打通 UART 通信”升级为“在 RA6M5 上接管手势识别主逻辑”。

当前版本需要同时完成两件事：

1. `uart2` 持续接收 STM32 手套端每 `20ms` 发来的一帧 10 维特征数据。
2. RA6M5 基于接收到的特征，按 `100ms` 更新一次分类样本，按 `200ms` 做一次 SVM 分类，并输出标签和命令结果。

当前设计仍然遵循“高内聚、低耦合”的分层原则：

- STM32 负责采集、滤波、特征整理、串口发帧。
- RA6M5 负责收帧、缓存、SVM 分类、结果输出。

## 2. 当前工程分层

### 2.1 HAL/FSP 自动生成层

关键目录：

- `uart/ra_gen/`
- `uart/ra_cfg/`
- `uart/ra/fsp/`

职责说明：

1. 提供 `g_uart2`、`g_uart7`、`g_ioport_ctrl` 等底层实例。
2. 提供 `R_SCI_UART_Open`、`R_SCI_UART_Read`、`R_SCI_UART_Write` 等 HAL 能力。
3. 提供 IO、时钟、中断等底层初始化入口。

说明：

- 这一层由 FSP 自动生成，不建议手工修改。

### 2.2 板级驱动层

关键文件：

- `uart/src/debug_uart/bsp_debug_uart.h`
- `uart/src/debug_uart/bsp_debug_uart.c`

职责说明：

1. 屏蔽 `g_uart2`、`g_uart7` 的直接使用。
2. 统一管理串口初始化、关闭、阻塞收发、回调事件。
3. 把 HAL 的异步读写行为封装成更稳定的阻塞接口。

这一层只关心“硬件串口怎么收发”，不关心“收发的数据属于什么业务”。

### 2.3 中间通信服务层

关键文件：

- `uart/src/debug_uart/mid_uart.h`
- `uart/src/debug_uart/mid_uart.c`

职责说明：

1. 在板级驱动层之上封装通用串口服务接口。
2. 提供字节流、字符串、浮点文本、浮点二进制、16 进制命令等通用能力。
3. 让协议层和业务层不需要直接面对 HAL 细节。

这一层只关心“串口数据如何组织与收发”，不关心具体协议语义。

### 2.4 手套协议层

关键文件：

- `uart/src/debug_uart/glove_frame.h`
- `uart/src/debug_uart/glove_frame.c`

职责说明：

1. 定义 STM32 -> RA6M5 的固定长度特征帧协议。
2. 负责在字节流中同步帧头 `0xAA 0x55`。
3. 负责版本号校验、CRC16 校验、小端解析。
4. 负责把 `int16` 定点特征还原为 `float` 特征数组。

这一层只关心“协议帧怎么收、怎么解”，不关心 SVM 分类逻辑。

### 2.5 算法模型层

关键文件：

- `uart/src/gesture/gesture_model.h`

职责说明：

1. 保存从 STM32 端迁移过来的 StandardScaler 参数。
2. 保存当前已经验证通过的线性 SVM 权重和偏置。
3. 提供纯 C 的 `GestureModel_Classify()` 分类接口。

这一层只关心“模型参数和推理公式”，不关心 UART、节拍、命令输出。

### 2.6 手势业务层

关键文件：

- `uart/src/gesture/gesture_service.h`
- `uart/src/gesture/gesture_service.c`

职责说明：

1. 基于连续收上来的特征帧维护 100ms 样本节拍。
2. 基于样本节拍维护 200ms 分类节拍。
3. 调用 `gesture_model.h` 完成 SVM 推理。
4. 将 `label` 映射成业务命令 `cmd`。
5. 统一输出调试文本，便于核对 RA 与 STM32 结果是否一致。

这一层只关心“如何用收上来的特征做分类业务”，不关心底层串口寄存器细节。

### 2.7 应用层

关键文件：

- `uart/src/hal_entry.c`

职责说明：

1. 初始化 LED、串口和手势业务层。
2. 持续从 `uart2` 收帧。
3. 把每一帧交给手势业务层决定是否更新样本、是否触发分类。
4. 当 200ms 分类结果就绪时，通过 `uart7` 输出标签和命令。

应用层现在只做流程编排，不直接处理 CRC、模型参数或命令映射表。

## 3. 当前数据链路

### 3.1 STM32 -> RA6M5

当前链路如下：

1. STM32 每 `20ms` 发送一帧固定长度二进制协议。
2. RA6M5 通过 `uart2` 持续阻塞接收。
3. 协议层完成帧头同步、CRC 校验和特征解析。
4. 手势业务层用合法帧更新业务节拍和分类输入。

### 3.2 RA6M5 内部节拍

当前 RA6M5 端不是“100ms 才去读一次串口”，而是：

1. 串口始终持续收包，避免 STM32 20ms 连续发帧时产生积压。
2. 每 `100ms` 从最近合法帧中更新一次分类样本。
3. 每 `200ms` 基于最新样本做一次 SVM 分类。

这样做的原因是：

1. 串口链路不会堵塞。
2. 分类节拍稳定。
3. 后续如果 STM32 提高发包频率，RA6M5 端结构仍然成立。

### 3.3 RA6M5 输出

当前输出方式：

- `uart7` 以文本形式打印：
  - `label`
  - `cmd`
  - 样本时间戳
  - 样本特征值

说明：

- 当前还没有绑定独立的“真实控制口”，因此命令结果先通过 `uart7` 调试输出验证。
- 后续如果你指定新的控制串口或总线，只需要替换业务层的结果派发方式，不需要改模型层和协议层。

## 4. 当前通信协议

协议总长度固定为 `31` 字节：

| 字段 | 字节数 | 说明 |
| --- | --- | --- |
| `sof1` | 1 | 固定 `0xAA` |
| `sof2` | 1 | 固定 `0x55` |
| `ver` | 1 | 协议版本，当前为 `0x01` |
| `seq` | 1 | 帧序号 |
| `tick_ms` | 4 | STM32 毫秒时间戳，小端 |
| `feature[10]` | 20 | 10 维特征，`int16` 小端，按 `float * 100` 定点化 |
| `status` | 1 | 状态字 |
| `crc16` | 2 | CRC16-CCITT-FALSE，小端 |

RA 端协议层输出两份特征：

- `feature_raw[10]`：原始 `int16`
- `feature[10]`：还原后的 `float`

## 5. 当前 SVM 输入定义

10 维特征顺序如下：

1. `f0x`
2. `f0y`
3. `f1x`
4. `f1y`
5. `f2x`
6. `f2y`
7. `f3x`
8. `f3y`
9. `roll`
10. `pitch`

这一顺序必须同时满足三端一致：

1. STM32 发包顺序
2. RA 协议解包顺序
3. `gesture_model.h` 的模型输入顺序

## 6. 当前命令映射

当前 `label -> cmd` 映射严格沿用 STM32 `control.c` 中的 `gr_cmd_map`：

| label | 手势 | cmd |
| --- | --- | --- |
| 0 | `fist` | `0x02` |
| 1 | `open` | `0x02` |
| 2 | `one` | `0x69` |
| 3 | `two` | `0x02` |
| 4 | `rock` | `0x05` |
| 5 | `up` | `0x08` |
| 6 | `down` | `0x0A` |
| 7 | `left` | `0x06` |
| 8 | `right` | `0x07` |

说明：

- 这里先追求“迁移后一致”，不在本阶段改命令语义。
- 如果后续你要重新定义命令协议，只改业务层映射表即可。

## 9. 当前主流程

`hal_entry()` 当前流程如下：

1. 初始化 LED。
2. 初始化 `uart2`、`uart7`。
3. 初始化手势业务层上下文。
4. 持续从 `uart2` 接收 STM32 发来的协议帧。
5. 协议层成功解帧后，交给业务层判断：
   - 是否到达 100ms 样本更新点；
   - 是否到达 200ms 分类点。
6. 当分类结果就绪时：
   - 翻转 LED；
   - 通过 `uart7` 输出 `label`、`cmd` 和当前特征快照。

## 11. 后续建议

接下来建议继续按下面顺序推进：

1. 在 RA 端记录一组“同一输入特征对应的 STM32 / RA 标签结果”，做一致性对比。
2. 如果结果一致，再决定是否把命令输出从 `uart7` 文本切换到独立控制口。
3. 如果后续需要更稳的识别结果，可以在业务层增加投票、滞回、命令保持等稳定策略。
4. 如果后续收包频率继续升高，可以把当前阻塞收包升级为“中断 + 环形缓冲区”。
5. 如果后续还要迁移挥手方向模型，可以继续在 `gesture/` 下新增独立业务模块，不要混入 `hal_entry.c`。

## 12. 当前注意事项

1. STM32 和 RA6M5 两端串口参数必须一致，例如波特率、数据位、停止位、校验位。
2. `uart2` 输入应保持纯二进制协议帧，不要混入调试文本。
3. 当前命令结果先走 `uart7` 文本调试输出，尚未绑定新的真实控制口。
4. 当前终端环境没有交叉编译工具链，因此代码修改后仍需在 e2studio 里重新编译确认。
5. `ra_gen` 和 FSP 自动生成文件不要手工修改，避免后续配置同步时被覆盖。
