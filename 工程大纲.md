# 工程大纲

## 1. 当前目标

当前 RA6M5 工程的目标是完成下面 3 件事：

1. `uart2` 持续接收 STM32 每 `20ms` 发来的一帧 10 维姿态特征数据。
2. RA6M5 基于接收到的特征，按 `100ms` 更新一次分类样本。
3. RA6M5 基于 `AGT0 10ms` 定时中断累计出 `200ms` 发送节拍，并在这个节拍上只发送原始 16 进制命令字节。

当前设计继续遵循“高内聚、低耦合”的分层原则：

- STM32 负责采集、滤波、特征整理、串口发帧。
- RA6M5 负责收帧、缓存、SVM 分类、命令过滤、定时发送。

## 2. 当前分层

### 2.1 HAL/FSP 自动生成层

关键目录：

- `uart/ra_gen/`
- `uart/ra_cfg/`
- `uart/ra/fsp/`

职责说明：

1. 提供 `g_uart2`、`g_uart7`、`g_timer_agt0`、`g_ioport_ctrl` 等底层实例。
2. 提供 `R_SCI_UART_Open`、`R_SCI_UART_Read`、`R_SCI_UART_Write`、`R_AGT_Open`、`R_AGT_Start` 等 HAL 能力。
3. 提供 IO、时钟、中断等底层初始化入口。

说明：

- 这一层由 FSP 自动生成，不建议手工修改。

### 2.2 串口板级驱动层

关键文件：

- `uart/src/debug_uart/bsp_debug_uart.h`
- `uart/src/debug_uart/bsp_debug_uart.c`

职责说明：

1. 屏蔽 `g_uart2`、`g_uart7` 的直接使用。
2. 统一管理串口初始化、关闭、阻塞收发、回调事件。
3. 把 HAL 的异步读写行为封装成更稳定的阻塞接口。

### 2.3 串口中间服务层

关键文件：

- `uart/src/debug_uart/mid_uart.h`
- `uart/src/debug_uart/mid_uart.c`

职责说明：

1. 在板级串口驱动层之上封装通用串口服务接口。
2. 提供字节流、字符串、浮点文本、浮点二进制、16 进制命令等通用能力。
3. 让协议层和应用层不需要直接面对 HAL 细节。

### 2.4 手套协议层

关键文件：

- `uart/src/debug_uart/glove_frame.h`
- `uart/src/debug_uart/glove_frame.c`

职责说明：

1. 定义 STM32 -> RA6M5 的固定长度特征帧协议。
2. 负责在字节流中同步帧头 `0xAA 0x55`。
3. 负责版本号校验、CRC16 校验、小端解析。
4. 负责把 `int16` 定点特征还原为 `float` 特征数组。

### 2.5 算法模型层

关键文件：

- `uart/src/gesture/gesture_model.h`
- `RS_bei/shoutao_you/tools/gesture_model.h`

职责说明：

1. `tools/train_svm.py` 负责把 CSV 训练结果导出成标准 C 头文件 `tools/gesture_model.h`。
2. RA6M5 侧的 `uart/src/gesture/gesture_model.h` 只做适配层，直接包含导出的模型文件。
3. STM32 和 RA6M5 共用同一份 StandardScaler 参数、SVM 权重、偏置和分类公式。
4. 后续重新训练模型时，不再需要把数组手工抄到 RA 工程里。

### 2.6 手势业务层

关键文件：

- `uart/src/gesture/gesture_service.h`
- `uart/src/gesture/gesture_service.c`

职责说明：

1. 基于连续收上来的特征帧维护 `100ms` 样本节拍。
2. 在 `200ms` 发送周期到来时执行一次分类周期。
3. 调用 `gesture_model.h` 完成 SVM 推理。
4. 将 `label` 映射成业务命令 `cmd`。
5. 复刻 STM32 端发送过滤逻辑：
   - `0x02` 不发送；
   - `0x00` 视为无动作，不发送；
   - 其他命令连续 `2` 次一致才允许发送。

### 2.7 定时器板级驱动层

关键文件：

- `uart/src/timer/bsp_dispatch_timer.h`
- `uart/src/timer/bsp_dispatch_timer.c`

职责说明：

1. 封装 FSP 自动生成的 `g_timer_agt0`。
2. 负责 `AGT0` 的 `open/start/stop`。
3. 在 `timer0_callback()` 中只向上层转发“10ms Tick 到了”。

### 2.8 定时器中间服务层

关键文件：

- `uart/src/timer/mid_dispatch_timer.h`
- `uart/src/timer/mid_dispatch_timer.c`

职责说明：

1. 把 `AGT0` 的 `10ms` 中断 Tick 聚合成 `200ms` 业务节拍。
2. 通过标志位向应用层暴露“现在可以发一次命令了”。
3. 应用层不直接接触 AGT0 和中断累计细节。

### 2.9 应用层

关键文件：

- `uart/src/hal_entry.c`

职责说明：

1. 初始化 LED、串口、手势业务层和定时器中间服务层。
2. 持续从 `uart2` 收帧。
3. 把每一帧交给手势业务层决定是否更新 `100ms` 样本。
4. 当 `200ms` 定时标志到来时，触发一次分类与命令派发判定。
5. `uart7` 当前只发送原始 16 进制命令字节，不发送任何文本调试信息。

## 3. 当前数据链路

### 3.1 STM32 -> RA6M5

当前链路如下：

1. STM32 每 `20ms` 发送一帧固定长度二进制协议。
2. RA6M5 通过 `uart2` 持续阻塞接收。
3. 协议层完成帧头同步、CRC 校验和特征解析。
4. 手势业务层用合法帧更新 `100ms` 分类输入。
5. 定时器中间层独立维护 `200ms` 发送节拍。

### 3.2 模型数据链路

当前模型链路如下：

1. `tools/collect_data.py` 采集手套动作数据并保存到 CSV。
2. `tools/train_svm.py` 基于 CSV 训练 SVM。
3. `tools/train_svm.py` 导出 `tools/gesture_model.h`。
4. RA6M5 侧的 `uart/src/gesture/gesture_model.h` 直接包含这个导出头文件。
5. `gesture_service.c` 调用导出文件里的 `GestureModel_Classify()` 完成推理。

### 3.3 RA6M5 内部节拍

当前 RA6M5 端不是“200ms 才去读一次串口”，而是：

1. 串口始终持续收包，避免 STM32 `20ms` 连续发帧时产生积压。
2. 每 `100ms` 从最近合法帧中更新一次分类样本。
3. `AGT0` 每 `10ms` 进入一次中断。
4. 定时器中间层累计 `20` 个 Tick 后置位一次 `200ms` 发送标志。
5. 主循环消费该标志后，基于最新样本做一次 SVM 分类与命令派发判定。

### 3.4 RA6M5 输出

当前输出方式：

- `uart7` 只发送原始命令字节，例如 `0x69`、`0x05`、`0x08`。

说明：

- 只有当命令通过过滤，并且到达 `200ms` 发送周期时，才会真正发一个原始字节。
- `0x02` 和 `0x00` 不会发出。
- 当前版本不再通过 `uart7` 输出任何文本调试信息。

## 4. 当前通信协议

协议总长度固定为 `31` 字节：

| 字段 | 字节数 | 说明 |
| --- | --- | --- |
| `sof1` | 1 | 固定 `0xAA` |
| `sof2` | 1 | 固定 `0x55` |
| `ver` | 1 | 协议版本，当前为 `0x01` |
| `seq` | 1 | 帧序号 |
| `tick_ms` | 4 | STM32 毫秒时间戳，小端 |
| `feature[10]` | 20 | 10 维特征，`int16` 小端，按 `float * 100` 定点化 |
| `status` | 1 | 状态字 |
| `crc16` | 2 | CRC16-CCITT-FALSE，小端 |

RA 端协议层输出两份特征：

- `feature_raw[10]`：原始 `int16`
- `feature[10]`：还原后的 `float`

## 5. 当前 SVM 输入定义

10 维特征顺序如下：

1. `f0x`
2. `f0y`
3. `f1x`
4. `f1y`
5. `f2x`
6. `f2y`
7. `f3x`
8. `f3y`
9. `roll`
10. `pitch`

这一顺序必须同时满足三端一致：

1. STM32 发包顺序
2. RA 协议解包顺序
3. `tools/gesture_model.h` 的模型输入顺序

## 6. 当前命令映射

当前 `label -> cmd` 映射严格沿用 STM32 `control.c` 中的 `gr_cmd_map`：

| label | 手势 | cmd |
| --- | --- | --- |
| 0 | `fist` | `0x02` |
| 1 | `open` | `0x02` |
| 2 | `one` | `0x69` |
| 3 | `two` | `0x02` |
| 4 | `rock` | `0x05` |
| 5 | `up` | `0x08` |
| 6 | `down` | `0x0A` |
| 7 | `left` | `0x06` |
| 8 | `right` | `0x07` |

说明：

- 这里先追求“迁移后一致”，不在本阶段改命令语义。
- 当前额外叠加了一层派发策略：
  - `0x02` 不发送；
  - `0x00` 不发送；
  - 其他命令必须连续 `2` 次结果一致才发送。

## 9. 当前主流程

`hal_entry()` 当前流程如下：

1. 初始化 LED。
2. 初始化 `uart2`、`uart7`。
3. 初始化手势业务层上下文。
4. 启动 `AGT0`，每 `10ms` 进入一次中断。
5. 持续从 `uart2` 接收 STM32 发来的协议帧。
6. 协议层成功解帧后，交给业务层判断是否到达 `100ms` 样本更新点。
7. 当 `200ms` 周期标志就绪时：
   - 执行一次分类；
   - 进行 `0x02` 屏蔽和两次确认过滤；
   - 如果满足发送条件，就通过 `uart7` 发送一个原始命令字节。

## 11. 后续建议

1. 先在 RA 端记录一组“同一输入特征对应的 STM32 / RA 命令结果”，做一致性对比。
2. 如果结果一致，再决定是否把 `uart7` 的原始字节输出切换到新的控制口。
3. 如果后续需要更稳的识别结果，可以在业务层继续增加投票、滞回、命令保持等稳定策略。
4. 如果后续收包频率继续升高，可以把当前阻塞收包升级为“中断 + 环形缓冲区”。
5. 如果后续重新训练模型，优先通过 `tools/train_svm.py` 导出新的 `tools/gesture_model.h`，不要再手工改 RA 里的模型数组。

## 12. 当前注意事项

1. STM32 和 RA6M5 两端串口参数必须一致，例如波特率、数据位、停止位、校验位。
2. `uart2` 输入应保持纯二进制协议帧，不要混入调试文本。
3. `uart7` 当前只发原始 16 进制命令字节，不再输出文本调试信息。
4. 当前业务层已经启用 `0x02` 屏蔽和两次确认过滤，如果后面要改阈值，优先改 `gesture_service.h` 中的宏。
5. `ra_gen` 和 FSP 自动生成文件不要手工修改，避免后续配置同步时被覆盖。
6. 如果你改了 `tools/train_svm.py` 的导出格式，RA 侧默认会直接受影响，因此建议保持 `GM_N_FEATURES=10`、`GM_N_CLASSES=9`。
