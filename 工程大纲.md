# 工程大纲

## 1. 工程目标

当前工程面向 RA6M5 端的串口接收与调试链路重构，目标是把 STM32 手套端发来的固定长度特征帧稳定接入到瑞萨开发板，并为后续的手势分类模块预留清晰的分层接口。

当前这版重点完成三件事：

1. `uart2` 作为协议接收口，接收 STM32 发来的 31 字节固定帧。
2. `uart7` 作为调试口，把解析后的字段和值打印出来，方便联调。
3. 在 HAL 之上继续分层，做到高内聚、低耦合，避免把协议细节直接写死在 `hal_entry.c`。

## 2. 当前分层结构

### 2.1 HAL/FSP 自动生成层

关键目录：

- `uart/ra_gen/`
- `uart/ra_cfg/`
- `uart/ra/fsp/`

职责说明：

1. 提供 `g_uart2`、`g_uart7`、`g_ioport_ctrl` 等底层实例。
2. 提供 `R_SCI_UART_Open`、`R_SCI_UART_Read`、`R_SCI_UART_Write` 等 HAL 能力。
3. 提供引脚、中断、时钟等底层配置。

说明：

- 这一层由 FSP 自动生成，不建议手工修改。

### 2.2 板级驱动层

关键文件：

- `uart/src/debug_uart/bsp_debug_uart.h`
- `uart/src/debug_uart/bsp_debug_uart.c`

职责说明：

1. 屏蔽 `g_uart2`、`g_uart7` 的直接使用。
2. 统一管理串口初始化、关闭、阻塞收发、回调事件。
3. 把 HAL 的异步回调语义封装成更好用的阻塞接口。

这一层只关心“硬件串口怎么收发”，不关心“收发的业务数据是什么”。

### 2.3 中间通信服务层

关键文件：

- `uart/src/debug_uart/mid_uart.h`
- `uart/src/debug_uart/mid_uart.c`

职责说明：

1. 基于板级驱动层，向上提供通用串口服务接口。
2. 支持原始字节、字符串、浮点文本、浮点二进制、16 进制指令等通用能力。
3. 让应用层和协议层不需要直接面对 HAL 细节。

这一层只关心“怎样组织串口数据”，不关心具体协议字段含义。

### 2.4 手套协议服务层

关键文件：

- `uart/src/debug_uart/glove_frame.h`
- `uart/src/debug_uart/glove_frame.c`

职责说明：

1. 定义 STM32 -> RA6M5 固定帧协议。
2. 负责在字节流中同步帧头 `0xAA 0x55`。
3. 负责校验协议版本和 CRC16。
4. 负责把 `int16` 定点特征恢复为 `float` 特征数组。
5. 负责把解析结果通过 `uart7` 打印成可读调试文本。

这一层只关心“手套协议怎么收、怎么解、怎么打印”，不关心后续手势分类算法。

### 2.5 应用层

关键文件：

- `uart/src/hal_entry.c`

职责说明：

1. 完成系统初始化。
2. 调用协议层从 `uart2` 收帧。
3. 调用协议层通过 `uart7` 打印调试信息。
4. 用 LED 翻转表示“已收到有效帧”。

应用层现在只做流程编排，不直接处理串口寄存器、CRC 细节和协议字段拆包。

## 3. 当前串口职责划分

### 3.1 uart2

用途：

- 接收 STM32 手套端发来的固定长度特征帧。

特点：

- 输入是二进制协议帧，不适合混入 `printf` 文本。
- 当前由 `GloveFrame_Receive()` 以阻塞方式逐字节同步帧头并完成整帧接收。

### 3.2 uart7

用途：

- 打印瑞萨端解析后的调试信息。

打印内容包括：

1. 帧序号 `seq`
2. 协议版本 `ver`
3. 时间戳 `tick_ms`
4. 状态字 `status`
5. CRC
6. 10 维特征值
7. 状态字对应的标志位名称

## 4. STM32 -> RA6M5 通信协议

当前协议与 STM32 端 `trans.h / trans.c` 保持一致。

### 4.1 帧格式

总长度固定为 `31` 字节：

| 字段 | 字节数 | 说明 |
| --- | --- | --- |
| `sof1` | 1 | 固定 `0xAA` |
| `sof2` | 1 | 固定 `0x55` |
| `ver` | 1 | 协议版本，当前为 `0x01` |
| `seq` | 1 | 帧序号 |
| `tick_ms` | 4 | STM32 侧毫秒时间戳，小端 |
| `feature[10]` | 20 | 10 维特征，`int16` 小端，按 `float * 100` 定点化 |
| `status` | 1 | 状态字 |
| `crc16` | 2 | CRC16-CCITT-FALSE，小端 |

### 4.2 特征顺序

当前 10 维特征顺序如下：

1. `mpu0.filter_angle_x`
2. `mpu0.filter_angle_y`
3. `mpu1.filter_angle_x`
4. `mpu1.filter_angle_y`
5. `mpu2.filter_angle_x`
6. `mpu2.filter_angle_y`
7. `mpu3.filter_angle_x`
8. `mpu3.filter_angle_y`
9. `jy901s.angle_roll`
10. `jy901s.angle_pitch`

RA 端在协议层内同时保留两份结果：

- `feature_raw[10]`：原始 `int16`
- `feature[10]`：还原后的 `float`

### 4.3 CRC 规则

CRC 采用：

- 算法：`CRC16-CCITT-FALSE`
- 多项式：`0x1021`
- 初值：`0xFFFF`
- 不反转输入输出
- 结果不再异或

CRC 覆盖范围是前 `29` 字节，即：

- 从 `sof1` 开始
- 到 `status` 结束
- 不包含最后的 `crc16` 本身

## 5. 当前调用链

瑞萨端接收链路如下：

1. `hal_entry.c`
2. `GloveFrame_Receive()`
3. `MID_Uart_ReceiveBytes()`
4. `BSP_DebugUart_ReadBlocking()`
5. `g_uart2.p_api->read()`

瑞萨端调试打印链路如下：

1. `hal_entry.c`
2. `GloveFrame_PrintDebug()`
3. `MID_Uart_SendString() / MID_Uart_SendBytes()`
4. `BSP_DebugUart_WriteBlocking()`
5. `g_uart7.p_api->write()`

## 7. 当前主流程

`hal_entry()` 当前流程如下：

1. 初始化 LED。
2. 初始化 `uart2`、`uart7`。
3. 通过 `uart7` 打印启动提示。
4. 持续等待 `uart2` 收到一帧合法手套数据。
5. 收到有效帧后翻转 LED。
6. 通过 `uart7` 打印这一帧的解析结果。
7. 继续等待下一帧。

## 8. 当前调试输出示例

`uart7` 打印风格示例：

```text

wait uart2 frame: AA 55 ... 31 bytes

mpu0_x=12.34  mpu0_y=-3.21
mpu1_x=11.98  mpu1_y=-2.88
mpu2_x=10.45  mpu2_y=-1.67
mpu3_x=9.90   mpu3_y=-0.88
roll=5.66     pitch=-8.77
flags=VALID|MODE1
```

## 9. 后续扩展建议

建议后面按下面方向继续扩展：

1. 在 `glove_frame` 之上新增 `gesture_service`，专门负责手势分类，不要把分类逻辑塞回 `hal_entry.c`。
2. 如果后续收包频率升高，建议把 `GloveFrame_Receive()` 从阻塞逐字节读取升级为“中断 + 环形缓冲区”方案。
3. 如果后续需要下发命令给 STM32，可以新增 `glove_command.h/.c`，不要把双向协议混进同一个文件里。
4. 如果后续需要记录训练样本，可以直接把 `feature_raw[]` 或 `feature[]` 导出到上位机。
5. 如果后续要做掉线检测，可以在应用层增加 `seq` 连续性检查和超时监测。

## 10. 当前注意事项

1. 当前实现默认 STM32 与 RA6M5 两端串口参数已经一致，例如波特率、数据位、停止位、校验位。
2. 当前 `uart2` 输入应保持纯二进制协议帧，不要混入调试文本。
3. 当前终端环境没有交叉编译工具链，因此代码已完成静态修改，但还需要你在 e2studio 里重新编译确认。
4. `ra_gen` 和 FSP 自动生成文件不要手工修改，避免下次配置同步时被覆盖。
