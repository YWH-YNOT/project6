# 工程大纲

## 1. 工程目标

本工程面向 RA6M5 平台的串口功能重构，目标是把原来直接写在应用层里的 `uart2`、`uart7` 调用整理成清晰的三层结构：

1. 板级驱动层
2. 中间服务层
3. 应用层

这样做的目的有三个：

1. 降低应用层和 HAL/FSP 的耦合
2. 提高 UART 功能复用性
3. 方便后续扩展协议解析、传感器通信和上位机调试

## 2. 当前工程分层

### 2.1 HAL/FSP 自动生成层

这一层由 Renesas FSP 自动生成，不建议手工修改。

关键文件：

- `uart/ra_gen/hal_data.h`
- `uart/ra_gen/hal_data.c`
- `uart/ra_gen/common_data.h`
- `uart/ra_gen/vector_data.c`

职责说明：

1. 提供 `g_uart2`、`g_uart7` 等底层实例
2. 提供中断向量和回调声明
3. 提供 IOPORT、IRQ 等基础外设配置

### 2.2 板级驱动层

关键文件：

- `uart/src/debug_uart/bsp_debug_uart.h`
- `uart/src/debug_uart/bsp_debug_uart.c`

职责说明：

1. 屏蔽 `g_uart2`、`g_uart7` 的直接使用
2. 统一管理串口初始化、关闭、发送、接收
3. 统一处理 `uart2_callback`、`uart7_callback`
4. 把 FSP 的异步读写封装成更易用的阻塞式接口

这一层只关注“硬件怎么收发”，不关心“发的是什么业务数据”。

### 2.3 中间服务层

关键文件：

- `uart/src/debug_uart/mid_uart.h`
- `uart/src/debug_uart/mid_uart.c`

职责说明：

1. 基于板级驱动层提供更易调用的业务接口
2. 支持字符串发送
3. 支持浮点数文本收发
4. 支持浮点数二进制收发
5. 支持 16 进制原始字节收发
6. 支持 16 进制字符串命令发送

这一层只关心“如何表达数据”，不关心具体串口寄存器和回调细节。

### 2.4 应用层

关键文件：

- `uart/src/hal_entry.c`

职责说明：

1. 完成系统启动后的应用初始化
2. 调用中间层接口发送浮点数和 16 进制命令
3. 处理按键触发、LED 指示等业务动作

应用层不直接调用 `g_uart2.p_api->write()` 之类的 HAL 接口，只通过 `MID_Uart_*` 系列函数完成业务操作。

## 3. 当前 UART 调用链

应用层调用路径如下：

1. `hal_entry.c`
2. `MID_Uart_*`
3. `BSP_DebugUart_*`
4. `g_uart2 / g_uart7`
5. `R_SCI_UART_*`

对应关系如下：

- 应用层发送文本浮点数：`MID_Uart_SendFloatText()`
- 应用层发送 16 进制命令：`MID_Uart_SendHexString()`
- 中间层最终调用：`BSP_DebugUart_WriteBlocking()`
- 板级层最终调用：`g_uartX.p_api->write()`

## 4. 当前支持的接口能力

### 4.1 板级驱动层接口

- `BSP_DebugUart_Init()`

- `BSP_DebugUart_WriteBlocking()`
- `BSP_DebugUart_ReadBlocking()`
- `BSP_DebugUart_ReadByteBlocking()`

### 4.2 中间服务层接口

- `MID_Uart_SendFloatText()`
- `MID_Uart_ReceiveFloatText()`

- `MID_Uart_ReceiveFloatBinary()`

## 5. 示例调用方式

### 5.1 发送文本浮点数

```c
MID_Uart_SendFloatText(MID_UART_PORT_7, 36.58f, 2U, true);
```

说明：

1. 通过 `uart7` 发送
2. 数据内容为 `36.58`
3. 保留 2 位小数
4. 自动追加 `\r\n`

### 5.2 发送 16 进制指令字符串

```c
MID_Uart_SendHexString(MID_UART_PORT_2, "AA 55 01 02 0D 0A");
```

说明：

1. 通过 `uart2` 发送
2. 中间层会把字符串解析成真实字节
3. 最终发送的是二进制命令，不是 ASCII 文本

### 5.3 接收文本浮点数

```c
float value = 0.0f;
MID_Uart_ReceiveFloatText(MID_UART_PORT_7, &value);
```

说明：

1. 中间层会自动跳过前导空格、回车、换行
2. 遇到分隔符后结束解析
3. 结果写入 `value`

### 5.4 接收二进制浮点数组

```c
float values[3];
MID_Uart_ReceiveFloatBinary(MID_UART_PORT_2, values, 3U);
```

说明：

1. 适用于 MCU 与 MCU 之间的高效传输
2. 默认按 `float` 的二进制布局接收

## 6. 当前目录建议

建议后续保持下面的结构继续扩展：

```text
uart/
├─ ra_gen/                  # FSP 自动生成层
├─ src/
│  ├─ debug_uart/
│  │  ├─ bsp_debug_uart.h   # 板级驱动层
│  │  ├─ bsp_debug_uart.c
│  │  ├─ mid_uart.h         # 中间服务层
│  │  └─ mid_uart.c
│  ├─ key/
│  ├─ led/
│  └─ hal_entry.c           # 应用层
└─ Debug/
```

## 7. 后续扩展建议

后续如果继续做姿态识别手套相关功能，建议按下面方向扩展：

1. 在中间层新增“协议帧解析模块”，不要把协议判断直接写进 `hal_entry.c`
2. 如果后续有陀螺仪、蓝牙、上位机三类串口业务，建议继续拆成 `service_xxx_uart.c`
3. 如果接收数据变长，建议在板级层增加环形缓冲区或 DMA 接收方案
4. 如果后续需要命令应答机制，建议在中间层增加“发送 + 等待响应”的事务接口

## 8. 当前注意事项

1. 当前实现以裸机阻塞式收发为主，适合现阶段调试和功能验证
2. 当前终端环境未提供交叉编译工具链，代码改动后还需要在 e2studio 中重新编译确认
3. `ra_gen` 下的 FSP 自动生成文件不要手工修改，避免下次配置同步时被覆盖
