# 工程框架

## 1. 分层结构

当前 `uart` 工程按照“板级驱动层 -> 中间层 -> 业务层 -> 应用层 -> 上位机工具层”组织。这样拆分的目的不是形式化分层，而是为了让每一层只关心自己该关心的事：

1. 板级驱动层只关心具体硬件资源怎么初始化、怎么收发、怎么进中断。
2. 中间层只负责把底层硬件能力重新包装成业务更容易理解的接口。
3. 业务层只处理“手势识别”和“手势采集导出”本身的逻辑。
4. 应用层只做启动、编排和模式切换，不深入掺杂算法细节。
5. 上位机工具层只负责落盘、标注、训练和导出模型。

## 2. 各层职责

### 2.1 板级驱动层

1. `uart/src/debug_uart/bsp_debug_uart.h`
2. `uart/src/debug_uart/bsp_debug_uart.c`
3. `uart/src/timer/bsp_dispatch_timer.h`
4. `uart/src/timer/bsp_dispatch_timer.c`
5. `uart/src/key/bsp_key_irq.h`
6. `uart/src/key/bsp_key_irq.c`
7. `uart/src/led/bsp_led.h`
8. `uart/src/led/bsp_led.c`

这一层只做四件事：

1. 把 `uart2`、`uart7` 的 `open/read/write/callback` 统一收口。
2. 把 `AGT0` 定时器的 `10ms Tick` 统一收口。
3. 把 `SW2` 外部中断转换成一次性锁存事件。
4. 把 LED 控制统一成简单宏接口。

这层故意不做业务判断。例如：

1. 串口层不会判断当前是“识别模式”还是“采集模式”。
2. 定时器层不会判断“现在是不是该分类”。
3. 按键层不会直接切模式。

## 2.2 中间层

1. `uart/src/debug_uart/mid_uart.h`
2. `uart/src/debug_uart/mid_uart.c`
3. `uart/src/debug_uart/glove_frame.h`
4. `uart/src/debug_uart/glove_frame.c`
5. `uart/src/timer/mid_dispatch_timer.h`
6. `uart/src/timer/mid_dispatch_timer.c`
7. `uart/src/key/mid_key_event.h`
8. `uart/src/key/mid_key_event.c`

这一层是“硬件语义 -> 业务语义”的转换层：

1. `mid_uart` 把底层串口驱动包装成“发字节”“发字符串”“发浮点文本”“发命令字节”。
2. `glove_frame` 把 uart2 上来的原始字节流同步成完整协议帧，再解成 `glove_frame_t`。
3. `mid_dispatch_timer` 把底层 `10ms Tick` 累加成业务真正关心的 `200ms` 发送节拍，并且只锁存一个待消费标志，不补账。
4. `mid_key_event` 把底层“中断触发”包装成“按键事件已发生”。

换句话说，应用层到了这一层以后，已经不需要再理解底层寄存器、回调事件和硬件实例名。

## 2.3 业务层

1. `uart/src/gesture/gesture_service.h`
2. `uart/src/gesture/gesture_service.c`
3. `uart/src/gesture/gesture_capture_service.h`
4. `uart/src/gesture/gesture_capture_service.c`
5. `uart/src/gesture/gesture_model.h`
6. `uart/tools/generated/gesture_model.h`

这层分成两条互斥链路：

1. 识别链路
   - 目标是“从当前样本得到一个稳定的控制命令”。
   - 处理步骤是：有效帧筛选 -> 50ms 采样 -> 100ms 分类 -> 命令映射 -> 二次确认过滤 -> 200ms 定时发送。
2. 采集链路
   - 目标是“把当前帧导出成可训练的 CSV”。
   - 处理步骤是：有效帧筛选 -> 10 列浮点文本输出。

这里的关键点是：识别链路和采集链路虽然共享底层输入 `glove_frame_t`，但对外输出完全不同，因此必须在应用层明确互斥。
另外，识别链路里关于 `0x19` 命令现在只保留一条路径：

1. “手势三”不再走规则旁路，而是作为 SVM 第 10 类标签 `three` 参与正常分类。
2. 只有当 `label=9` 被映射成 `0x19`，并且通过现有二次确认过滤后，应用层才会更新当前激活命令。
3. STM32 协议帧里的 `KEY1/KEY2` 状态位当前仅作为协议数据保留，不再参与 RA6M5 侧命令派发。

## 2.4 应用层

1. `uart/src/hal_entry.c`

应用层只做三类事情：

1. 启动初始化。
2. 主循环流程编排。
3. 模式切换。

它不会自己解协议，不会自己做 SVM 分类，也不会自己维护 50ms/100ms/200ms 这些节拍计数。
它只负责维护“当前激活命令”，并把这个激活命令与每个 `200ms` 发送窗口配对起来。

## 2.5 上位机工具层

1. `uart/tools/collect_data.py`
2. `uart/tools/train_svm.py`
3. `uart/tools/tool_config.py`
4. `uart/tools/data/gesture_data.csv`
5. `uart/tools/generated/gesture_model.h`

这层负责板外闭环：

1. 从 `uart7` 读取 CSV。
2. 把 CSV 带标签保存到本地。
3. 在保留旧数据的前提下，为 `three(label=9)` 继续补采样本。
4. 用本地数据训练线性 SVM。
5. 把训练结果同时导出成工具侧留档头文件和固件实际使用的头文件。

## 3. 固件启动流程

系统启动后，主流程是固定的：

1. `R_BSP_WarmStart()` 在启动早期被调用。
2. 在 `RESET` 阶段打开数据 Flash 访问能力。
3. 在 `POST_C` 阶段完成引脚复用配置。
4. 进入 `hal_entry()`。
5. 初始化 LED、UART、按键事件层和基于 `AGT0 10ms` 的双节拍调度定时器。
6. 默认进入识别模式。
7. 主循环开始持续运行。

## 4. 主循环运行逻辑

`hal_entry()` 主循环每一轮都按下面顺序工作：

1. 先检查一次是否有新的按键事件。
2. 阻塞等待 `uart2` 收到一帧新的合法手套数据。
3. 收到后再补检查一次按键事件。
4. 如果当前是采集模式，就直接把这帧导出成 CSV。
5. 如果当前是识别模式，就先尝试刷新 50ms 样本缓存。
6. 如果 100ms 分类节拍到点，就执行一次 SVM 分类，并更新当前激活命令。
7. 如果 200ms 发送节拍到点，并且当前确实存在激活命令，就从 `uart7` 重复发出 1 字节命令。

这个顺序的原因很重要：

1. 先处理按键，是为了让模式切换响应足够快。
2. 先收完整帧，再决定走哪条业务链路，是为了保证采集和识别都建立在同一份合法数据上。
3. 采样和分类拆开，是为了降低抖动，避免每一帧都直接驱动命令输出。

## 5. 识别链路详细流程

识别模式下，一帧数据并不会立刻变成命令，而是要经过下面几层处理：

### 5.1 协议层处理

1. `GloveFrame_Receive()` 持续从 `uart2` 读字节。
2. 找到帧头 `AA 55`。
3. 收满固定长度 `31` 字节。
4. 校验版本号和 CRC。
5. 解出 `seq`、`tick_ms`、`status` 和 10 维特征。

### 5.2 50ms 采样

1. `GestureService_ProcessFrame()` 只接收 `DATA_VALID` 的帧。
2. 业务层不再根据协议状态位触发 `0x19`，当前帧只用于维护识别样本。
3. 如果当前还没有样本，立即保存为第一份样本。
4. 如果距离上次采样已经过了 `50ms`，就用当前帧刷新样本缓存。
5. 如果 50ms 还没到，就保留旧样本，不更新。

这一步的目的，是把高速连续帧变成较稳定的分类输入节拍。

### 5.3 100ms 分类

1. `MID_DispatchTimer` 把底层 `10ms Tick` 累加成 `100ms` 分类周期和 `200ms` 发送周期。
2. 当应用层轮询到新的 100ms 分类标志时，调用 `GestureService_RunClassifyCycle()`。
3. 如果本轮还没有新的样本，就跳过普通 SVM 分类，避免对同一份旧样本重复分类。
4. 如果有新样本，就调用 `GestureModel_Classify()` 得到标签。
5. 如果后续通过 `tools/train_svm.py` 重新训练出了 `three(label=9)`，它会在这里按普通标签映射到 `0x19`。
6. 如果分类结果已经通过过滤，应用层会把这个命令更新成当前激活命令。
7. 如果下一轮只是短暂抖动到等待确认状态，应用层仍会继续沿用旧命令，避免发送断断续续。

### 5.4 200ms 发送

1. `MID_DispatchTimer` 另外还会独立给出 `200ms` 发送标志。
2. 应用层只在这个发送标志到点时才允许真正往 `uart7` 发 1 字节命令。
3. 只要当前激活命令还有效，每个 `200ms` 发送窗口都会重复发送同一个命令字节。
4. 如果主循环错过了多个发送周期，中间层也不会补发旧节拍，因此发送频率始终被限制为“至少 0.2 秒一次”。

### 5.5 命令映射与过滤

1. 标签先映射成控制命令值。
2. 如果命令是 `0x02` 或 `0x00`，直接压制，不发送。
3. 如果命令与上一次候选命令相同，确认次数加一。
4. 只有当同一命令连续达到 `2` 次，才真正允许进入“待发送”状态。
5. “手势三”走的是普通 SVM 标签分支，因此仍然参与两次确认过滤。
6. 应用层看到 `send_ready == true` 后，会把 `send_cmd` 更新成当前激活命令。
7. 如果中间出现一次 `send_hold`，应用层不会立刻停发旧命令，而是等新命令稳定或无效结果连续出现后再切换。

这一步的目的，是减少抖动和误触发。

## 6. 采集链路详细流程

采集模式下，主循环不会再做分类，而是把数据导出给上位机：

1. `GloveFrame_Receive()` 仍然从 `uart2` 收合法协议帧。
2. `GestureCaptureService_ProcessFrame()` 只导出 `DATA_VALID` 的帧。
3. 每帧输出 10 个浮点值。
4. 输出格式固定为一行：

`f0x,f0y,f1x,f1y,f2x,f2y,f3x,f3y,roll,pitch`

5. 输出串口固定为 `uart7`。
6. `collect_data.py` 从 `uart7` 按行读取并保存到本地 CSV。

## 7. 模式切换逻辑

模式切换由 `SW2` 触发，但真正切换动作发生在主循环里：

1. 外部中断只锁存一次“按下事件”。
2. 主循环通过 `MID_KeyEvent_TryConsumeKey1Press()` 消费这个事件。
3. 如果原来是识别模式，就切到采集模式。
4. 如果原来是采集模式，就切回识别模式。
5. 每次切换时都会：
   - 清空识别业务状态；
   - 清空应用层当前激活命令；
   - 清空 100ms/200ms 调度节拍；
   - 更新 LED 状态。

LED 含义如下：

1. LED 熄灭：当前是识别模式。
2. LED 常亮：当前是采集模式。
3. LED 翻转一下：识别模式下刚刚成功发出一个控制命令。

## 8. 模型更新闭环

当前模型更新流程已经完全闭环，不需要手工改固件权重：

1. 烧录 `uart` 工程。
2. 按 `SW2` 进入采集模式。
3. 在 `uart/tools` 运行：

```bash
python collect_data.py --port COM7
```

4. 采完数据后运行：

```bash
python train_svm.py
```

5. 训练脚本会生成：

`uart/tools/generated/gesture_model.h`

6. 训练脚本还会同步覆盖：

`uart/src/gesture/gesture_model.h`

7. 重新编译 `uart` 工程后，新的模型就会生效。

## 9. 当前关键约束

1. 协议层当前固定输出 10 维特征。
2. `tools` 训练链已经升级为 10 类标签，其中 `label=9` 对应 `three`。
3. 当前工具侧和固件侧模型头文件都已经升级为 10 类版本，其中 `label=9` 对应 `three`。
4. 采集模式和识别模式必须严格互斥。
5. `uart7` 在任意时刻只能承担一种输出职责：
   - 要么输出 1 字节命令；
   - 要么输出 CSV 文本。

这就是当前工程最核心的运行约束。如果以后要继续扩展，比如同时加调试文本输出、实时波形输出或更多按键行为，都必须先考虑这几条约束是否会被打破。
